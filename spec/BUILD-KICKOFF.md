## Non-negotiable rules — a violation is a build defect, not a style choice
1. **BSV-only, post-Genesis.** No BTC code, no BTC assumptions.
   `OP_CHECKLOCKTIMEVERIFY`/`OP_CHECKSEQUENCEVERIFY` are **NO-OPS** — enforce ALL timing at
   the **transaction level** (`nLockTime` + `nSequence` under the original replacement rule),
   never in-script.
2. **OP_RETURN is BANNED everywhere.** Every commitment/anchor is push-data in a **live**
   script (`<data> OP_DROP` prefix, or an unlocking-script push). Any `OP_RETURN` output is a
   reject. Add a CI/lint check that fails the build if the `OP_RETURN` opcode (0x6a) appears
   in any locking or unlocking script.
3. **Zero fabrication.** Every number, vector, and byte size is **generated by running code**
   and committed as a reproducible vector — never written from memory. A `reproduce` command
   regenerates all of them and exits non-zero on any mismatch.
4. **Real-interpreter tests.** Script spends run through the real BSV Script interpreter with
   **Genesis rules**; negative tests MUST fail **inside** the interpreter, never in a wrapper
   guard; signature spot-checks are not an acceptable substitute.
5. **No overclaim; enumerate the trust surface. No hidden assumptions** — record decisions as
   ADRs. **No apologies.**
6. **Traceability is complete.** Every requirement (core `REQ-*` and app `REQ-APP-*`) maps to
   code and a passing test; CI fails on any untested requirement or untraced consensus/security
   source file.
7. **Engineering bar:** NASA NPR 7150.2 assurance practice + a **documented** Power-of-Ten
   adaptation (rule 3 and the raw-pointer rules are N/A in a GC TS/Go runtime — say so) +
   Microsoft SDL.

## Build order — commit at each gate
Per the phased roadmap (core §17, app §A21.9):
- **Phase 0 — foundations.** Scaffold the monorepo (core §16 / app §A2.4): `/spec`,
  `/packages/*` (protocol-types, engine, hand-eval, game-*, crypto-mentalpoker,
  script-templates-ts, tx-builder, wallet-custody, adapters, sdk, ui-core, app-services),
  `/apps/{client-web,client-desktop,relay-go,indexer-go}`, `/vm`, `/tests`. Adapters +
  conformance-bound fakes for CT/BS/VA/OB. The VM brings up node(regtest)+relay+empty client;
  `reproduce` green; traceability skeleton live. **Gate:** VM launches end-to-end, self-test
  passes, CI all green. **Commit and tag.**
- **Phase 1 — first playable: heads-up NL Texas Hold'em on regtest, Windows + web, with
  discovery.** Entropy commit/reveal; distributed shuffle; encrypted-card deal; full
  preflop→river betting FSM; minimum-reveal showdown; settlement; decision + recovery
  timeouts; **fold without reveal**; relay + LAN discovery; the Tauri Windows shell and the
  web shell over one shared UI core; transcript + deterministic replay. **Generate the §19.D
  vectors by running `handeval_oracle.py`.** Run the interpreter-level script tests (Genesis),
  the adversarial subset, the E2E in the VM, and `reproduce`. **Gate:** core §14.7 / app
  §A18.3, on **both** desktop and web builds. **Commit and tag.**
- **Phase 2+ and the other games** (Omaha, Stud, Draw, Razz; then Blackjack after its §A21.7
  distinct dealerless model is fixed) per app §A21.9 — each a `GameModule`, each with its
  variant table-view profile, **generated** hand-eval vectors, betting structures (PL/FL), and
  its acceptance gate. **Commit per game.**

## Run & build obligations — you execute these
- Generate vectors: `python3 handeval_oracle.py` → embed verified output; `reproduce` checks
  it. Extend the oracle for new vectors; never hand-write a vector.
- Build and **measure** the Script templates through the real interpreter (core §19.C); commit
  the byte schedules as reproducible vectors.
- Build the self-contained VM image (container; optional VM image) with a one-command
  bootstrap (core §10, app §A14).
- Build the **signed Windows installer (Tauri)** and the **web bundle** from the same commit;
  record artifact hashes (app §A14).
- CI stages (app §A14.2): typecheck → lint (incl. the OP_RETURN-absence check) → unit+property
  → interpreter-level (Genesis) → integration → build image → E2E-in-image → `reproduce` →
  accessibility + security → traceability. A red stage blocks merge.
